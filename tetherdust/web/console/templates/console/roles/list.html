Roles & Permissions

What are Roles?

Roles are the access control units in TetherDust. Every user is assigned exactly one role, which determines what tools, databases, and documentation sources they can access in the chat interface. Without a role, a user has no capabilities.

Roles are evaluated at the time a chat session begins — changing a role takes effect on the next session, not mid-conversation.

Tool Access

Specify which MCP tools users in this role can invoke. A tool must also be globally enabled (Tools section) to be usable — role access alone is not sufficient.

At minimum, most roles need list_tables and query_database. Add search_docs and get_table_schema for roles that benefit from documentation-assisted querying.

Database Access

Specify which database connections are accessible to users in this role. Users can only query databases their role explicitly permits. This is enforced inside the query_database tool before any SQL is executed.

Documentation Access

Specify which documentation sources appear in search_docs results for users in this role. Use this to restrict sensitive internal documentation to appropriate roles while keeping general documentation widely available.

Row Limit

The maximum number of rows the query_database tool will return for users in this role. This controls result size for both performance and data governance.

  • Set a low limit (e.g. 100) for general users to prevent accidental bulk exports.
  • Set a higher limit for power users or analysts who need full datasets.
  • Set to 0 for no limit — use only for trusted admin roles.

Manage Users

If enabled, users in this role can access the Users section of Mission Control to create, edit, and deactivate other user accounts. This does not grant access to any other admin panel sections — it is a narrow elevated permission for team leads or HR administrators.

Designing Your Role Structure

  • Start with a least-privileged Read-Only base role covering only the tools and databases most users need.
  • Create elevated roles (e.g. Analyst, Power User) that add more databases or higher row limits.
  • Reserve admin panel access for a dedicated Admin role with staff status.
Tip: Keep roles small and purposeful. It is easier to grant additional access than to audit over-privileged roles later.
