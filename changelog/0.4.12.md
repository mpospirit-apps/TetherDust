# 0.4.12 — 2026-06-26

**Backend fast-follow #3 (organize `settings.py`).** Restructure
`backend/project/settings.py` into clearly delimited, banner-headed sections for
readability. No runtime behavior change — every setting and value is preserved.

## Upgrade notes for admins
- None. No DB migrations, no new/renamed env vars, no value changes. Purely a
  source reorganization of `settings.py`.

## Changed
- **Grouped settings under 77-char banner section headers** (`SECURITY`,
  `LOGGING`, `APPLICATION DEFINITION`, `URLS AND WSGI`, `TEMPLATES`, `AUTH`,
  `DATABASE`, `INTERNATIONALIZATION`, `STATIC FILES`, `CELERY`, `CHANNELS`,
  `REST FRAMEWORK`, `TETHERDUST`) and dropped boilerplate "what this does"
  comments, keeping only substantive project-specific rationale.
- **`INSTALLED_APPS`** is sub-grouped (`Daphne` / Django core / Third-party /
  Project apps), and the four project apps now use their explicit dotted
  `AppConfig` paths (`engine.apps.EngineConfig`, `workspace.apps.WorkspaceConfig`,
  `management.apps.ManagementConfig`, `api.apps.ApiConfig`) instead of bare app
  labels. Behavior-equivalent — Django already resolved each app's single
  `AppConfig` — just explicit now.

## Notes
- Two deliberate deviations from a textbook section order, both to preserve
  correctness:
  - **`daphne` stays first** in `INSTALLED_APPS` (it must precede
    `django.contrib.staticfiles` for its ASGI `runserver` to win).
  - **`LOGGING` sits after `SECURITY`** (not in the conventional second slot)
    because `LOG_LEVEL` derives from `DEBUG`, which `SECURITY` defines.
- Verified: `manage.py check` clean, and a startup smoke confirms all apps load
  and the reliable-signal receiver from 0.4.11 is still connected.
