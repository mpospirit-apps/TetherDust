# 0.4.11 — 2026-06-26

**Backend fast-follow #2 (reliable report-result cleanup).** Move the
`ReportExecution` result-file cleanup off the synchronous `post_delete` path onto
a reliable, transaction-aware async signal, so a crash after the database commits
can no longer leak result directories on disk.

## Upgrade notes for admins
- No DB migrations, no new env vars, no manual steps. Rebuild the `backend`
  image (`docker compose up --build`). The cleanup now runs on the existing
  Celery worker — no new service.

## Added
- **`backend/project/signals.py` — a `ReliableSignal` base.** Receivers are
  scheduled inside the emitting database transaction (via
  `transaction.on_commit`) and run later as Celery tasks
  (`_dispatch_reliable_receiver`). If the transaction rolls back the task is
  never enqueued; if it commits the task is guaranteed queued. Delivery is
  at-least-once, so receivers must be idempotent.
- **`backend/engine/receivers.py` — `on_report_execution_deleted`.** Idempotent
  receiver that removes the on-disk result directory via the existing
  `engines.result_storage.delete_results` (which no-ops when the directory is
  already gone, so duplicate delivery is harmless).

## Changed
- **`backend/engine/signals.py`** — the `post_delete` `cleanup_report_results`
  receiver is now a thin **bridge**: when the execution recorded a result file it
  emits `report_execution_deleted.send_reliable(...)` with the execution id
  instead of deleting files synchronously inline. Emitting from `post_delete`
  keeps coverage of every delete path (API, cascade, admin) while running inside
  the delete's transaction.
- **`backend/engine/apps.py`** — `ready()` now imports `receivers` alongside
  `signals` so the reliable receiver is connected at startup.

## Notes
- `create_user_profile` (the `post_save` on `User`) intentionally stays a
  standard synchronous receiver — profile creation must happen in-band, not async.
- Implementation note: `ReliableSignal.send_reliable` reads Django's private
  `Signal._live_receivers`, which on **Django 6** returns a `(sync, async)` pair
  of receiver lists. Re-check this on any Django upgrade.
- Verified at startup (no DB needed): the Celery task registers, the signal is a
  `ReliableSignal`, and `engine.receivers.on_report_execution_deleted` is
  discovered as a live receiver.
