# 0.4.10 — 2026-06-26

**Backend fast-follow #1 (DRF central exception handler).** Map plain Python
exceptions to HTTP responses in one place instead of re-wrapping errors in each
view.

## Upgrade notes for admins
- No DB migrations, no new env vars, no manual steps. Rebuild the `backend`
  image (`docker compose up --build`) to pick up the change.

## Added
- **`backend/api/exceptions.py` — a central DRF `EXCEPTION_HANDLER`.** Plain
  exceptions raised below the view layer now map to clean HTTP responses with a
  `{"detail": "..."}` body:
  - `ValueError` → **400**
  - `LookupError` → **404**
  - `PermissionError` → **403**

  DRF's default handler runs first, so `APIException`, `Http404`,
  `PermissionDenied`, and auth failures keep their existing behavior; anything
  unmapped still becomes a `500`. This lets views/services `raise ValueError(...)`
  for bad input and get a `400` instead of an opaque `500`.

## Changed
- **`backend/project/settings.py`** — wired the handler via
  `REST_FRAMEWORK["EXCEPTION_HANDLER"] = "api.exceptions.custom_exception_handler"`.

## Notes
- Additive only — no existing endpoint behavior changes. The current per-view
  `try/except Exception` blocks (e.g. `ChartDataView`) return their own custom
  `500` JSON, which the new handler does **not** subsume, so they were left in
  place intentionally.
- Caveat by design: `LookupError` is the base class of `KeyError`/`IndexError`,
  so an *accidental* `KeyError` in a code path that reaches the handler will now
  surface as a `404`. Acceptable trade-off to enable the "raise `LookupError` for
  a missing record → 404" convention going forward.
