Database Connections

What are Database Connections?

Database connections are the data sources that TetherDust exposes to AI agents via the query_database and list_tables MCP tools. Each connection stores all the information needed to reach a database — agents never see or handle raw credentials directly.

You can register as many connections as needed. Role-based access control then determines which users can query which databases.

Supported Engines

  • PostgreSQL — Full support via psycopg2. Recommended for production.
  • MySQL — Full support via mysqlclient.
  • MariaDB — Full support via pymysql.
  • Microsoft SQL Server — Via pyodbc. Requires ODBC driver installed on the host.
  • Oracle — Via cx_Oracle. Requires Oracle Instant Client.
  • SQLite — File-based, no server required. Useful for testing and local datasets.
  • ClickHouse — Via clickhouse-connect. Columnar OLAP engine for analytics workloads.
  • Snowflake / Google BigQuery — Cloud warehouses via their respective SQLAlchemy dialects.

Connection Fields

  • Name — A human-readable label used in chat responses, audit logs, and role configuration. Choose something descriptive (e.g. "Production Orders DB").
  • Host / Port — The database server address and port. For SQLite, leave host blank and set the database field to the file path.
  • Database — The schema or catalog name to connect to. For SQLite this is the file path.
  • Username / Password — Credentials stored encrypted at rest. They are never exposed in logs or API responses.
  • Extra Options — Raw SQLAlchemy connection arguments as JSON (e.g. {"connect_timeout": 10, "sslmode": "require"}). Refer to your engine's SQLAlchemy docs for available keys.

Read-Only Enforcement

All queries are validated against a regex blocklist before execution. INSERT, UPDATE, DELETE, DROP, CREATE, ALTER, TRUNCATE, and other write or DDL statements are rejected and never reach the database server.

For maximum safety, also configure the database user with read-only grants on the server side. This provides a second layer of protection independent of TetherDust's own enforcement.

Testing Connections

Click the Test button next to any connection to verify reachability and credentials. The result shows round-trip latency and any driver-level error messages. Always test after adding a new connection or changing credentials.

Active vs Inactive

Inactive connections are excluded from the list_tables tool response, so agents will not discover or query them. Use this to temporarily disable a database without deleting its configuration.

Tip: Create a dedicated read-only database user for TetherDust rather than using an admin account. This adds a second layer of protection on top of TetherDust's own enforcement.
