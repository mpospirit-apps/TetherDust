# Installation

This page walks you through getting TetherDust running from a fresh clone to a
working stack. The only supported runtime is **Docker Compose** — all services
(PostgreSQL, Redis, MCP server, agent gateways, and the Django web app) run as
containers.

## Prerequisites

| Requirement | Notes |
|---|---|
| **Docker** | Version 24+ recommended. [Get Docker](https://docs.docker.com/get-docker/) |
| **Docker Compose v2** | Ships with Docker Desktop. Verify: `docker compose version` |
| **An AI agent credential** | One of: ChatGPT Plus/Pro subscription token, Claude Pro/Max OAuth token, OpenAI API key, or Anthropic API key. You configure this *after* first boot — it is not needed to start the stack. |

## Configure secrets

All configuration lives in a gitignored `.env` file that Docker Compose reads
automatically. Start from the template:

```bash
cp .env.example .env
```

Open `.env` and fill in the following — these are the only values the stack
**cannot** start without.

### a. Encryption key

All stored database passwords and agent credentials are encrypted with a
[Fernet](https://cryptography.io/en/latest/fernet/) key. Generate one:

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Set `TETHERDUST_ENCRYPTION_KEY` to the output. Keep this key secret and back it
up — losing it means stored credentials cannot be decrypted.

> In production (`DJANGO_DEBUG=false`), TetherDust refuses to save a credential
> if this key is missing. In development mode it logs a warning and stores
> plaintext — convenient locally, never acceptable for real deployments.

### b. Django secret key

Used for session signing, CSRF tokens, and cookie integrity:

```bash
python -c "import secrets; print(secrets.token_urlsafe(64))"
```

Set `DJANGO_SECRET_KEY` to the output.

### c. Admin credentials

The superuser is bootstrapped automatically on first boot from these values.
Change `admin`/`admin` to something real:

```
DJANGO_SUPERUSER_USERNAME=admin
DJANGO_SUPERUSER_PASSWORD=<strong-password>
DJANGO_SUPERUSER_EMAIL=you@example.com
```

### d. Database credentials (optional)

`DB_NAME`, `DB_USER`, and `DB_PASSWORD` default to `tetherdust`/`tetherdust`
in `.env.example`. These are fine for local use; change them for any exposed
deployment. A single set of variables feeds all services that need them — you
do not have to repeat them in multiple places.

## Launch the stack

```bash
docker compose up --build
```

On first run Docker builds the images (a few minutes). On subsequent starts
`--build` can be omitted. The stack is ready when you see the Django startup
message:

```
web-1  | Starting TetherDust dev server on port 8000 (auto-reload enabled)...
```

**What happens on first boot** (in `docker-entrypoint.sh`):

1. Waits for PostgreSQL to become healthy.
2. Runs `manage.py migrate` to create all tables.
3. Creates the superuser from `DJANGO_SUPERUSER_*` env vars.
4. Auto-discovers documentation sources by scanning the `documentations/`
   directory and creating `DocumentationSource` records for each folder.
5. Ensures every user has a `UserProfile` and assigns the built-in `Admin` role
   to any superuser that does not already have one.

---

## First login

Open **http://localhost:8000** and log in with the superuser credentials you set
in step c. You land on the workspace home page; the admin management is at
**http://localhost:8000/management/**.

---

## Connect an agent

From the management, go to **Agents** and add an `AgentConfiguration`:

1. Choose the agent type (e.g. `claude_code` for Claude Pro/Max OAuth).
2. Paste your credential (auth token or API key).
3. Mark it **Active**.

Only one agent is active at a time. Switching is a database change — no
restart needed. See
[[TetherDust Documentation/3. Agent Integrations/1. Overview.md|Agent Integrations]]
for the full breakdown of agent types and credentials.

---

## Connect a database

From the management, go to **Databases** and add a `DatabaseConnection`:

1. Choose the engine (PostgreSQL, MySQL, ClickHouse, etc.).
2. Enter host, port, database name, and credentials for a **read-only** account.
3. Save. The MCP server picks up the connection immediately — no restart.

> **Always use a read-only database account.** This is the primary safeguard
> against the agent accidentally modifying data. See
> [[TetherDust Documentation/5. Security/2. Read-Only Enforcement.md|Read-Only Enforcement]]
> for the full three-layer protection model.

Once you have an agent and a database configured, open the chat and ask a
natural-language question to verify everything is working.

## What starts automatically

`docker compose up` starts ten services by default:

| Service | Port (in-stack) | Purpose |
|---|---|---|
| `db` | 5432 | PostgreSQL 16 — the app database |
| `mcp` | 8001 | Built-in MCP server (database tools, doc search) |
| `local-mcp` | 8003 | Local custom MCP server (user-defined tools) |
| `codex` | 8002 | Codex gateway — subscription token auth |
| `codex-api` | 8002 | Codex gateway — OpenAI API key auth |
| `claude` | 8002 | Claude Code gateway — subscription OAuth auth |
| `claude-api` | 8002 | Claude Code gateway — Anthropic API key auth |
| `redis` | 6379 | Redis 7 — Celery broker and result backend |
| `web` | **8000** | Django app (the only port exposed to the host) |
| `celery-worker` | — | Celery worker (report execution, chart refresh) |
| `celery-beat` | — | Celery beat scheduler |

The agent gateways (`codex`, `codex-api`, `claude`, `claude-api`) all start by
default even if not configured — idle cost is one FastAPI process (~50–80 MB
each). This means switching the active agent is always a zero-downtime database
flip. The only service behind a Compose profile is `ollama` (local LLM):

```bash
# Start with local LLM support
docker compose --profile ollama up --build
```

Only the `web` service exposes a port to the host (`8000`). All inter-service
traffic stays on the Docker Compose network.
