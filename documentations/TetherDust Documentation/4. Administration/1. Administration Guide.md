## Console overview

The management is the staff-only administration surface at `/management/`. It requires
`is_staff = True` on the `auth.User` record. The superuser created at first boot
always has staff access.

The management sidebar is divided into sections:

| Section | What it manages |
|---|---|
| **Home** | Activity dashboard — query counts, recent queries, system health at a glance |
| **Chat** | Browse and inspect all users' chat sessions (read-only) |
| **Agents** | Agent configurations — credentials, system prompt, active agent |
| **Databases** | Database connections — add, edit, test |
| **Docs** | Documentation sources — add, edit, generate content with AI |
| **Codebases** | GitHub repository connections — add, sync |
| **MCP Servers** | Custom MCP server configurations (remote URLs and local subprocess MCPs) |
| **Reports** | Report definitions and execution history |
| **Dashboards** | Dashboard and chart management |
| **Tethers** | Tether definitions and graph versions |
| **Roles** | Role creation and permission assignment |
| **Users** | User accounts (requires `can_manage_users` or superuser) |
| **Audit** | Query audit log, doc generation log, chart generation log |
| **Settings** | Application-level settings |

## User management

User management requires `is_staff = True` **and** either `is_superuser = True`
or a role with `can_manage_users = True`.

In the normal management workflow, a role marked **Admin role** makes assigned
non-superusers staff. A role with **Can manage users** but without **Admin role**
does not grant management access by itself; it only matters for users who are
already staff by another path.

### Creating a user

1. Go to **Console → Users → New User**.
2. Set username, email, and password.
3. Optionally assign a **Role** from the dropdown.
4. Assign a role marked **Admin role** if the user should have management access.

A `UserProfile` is created automatically and linked to the chosen role. Admin
roles sync non-superuser accounts to `is_staff=True`.

### Editing a user

1. Go to **Console → Users**, click a user's name.
2. Change their role. Staff status follows whether the assigned role is marked
   **Admin role**.
3. Save. The change takes effect on the user's next request — no logout required.

### Deleting a user

Users can be deleted from Django's built-in admin (`/admin/auth/user/`). Chat
sessions are deleted with the user (cascade). Query audit log entries retain a
`null` user FK (set-null on delete).

### Password resets

Staff can set any user's password from Django admin (`/admin/auth/user/<id>/password/`).

## Role management

See [[TetherDust Documentation/2. Features/11. Roles & Access Control.md|Roles & Access Control]]
for the full permission model. The short workflow:

1. **Console → Roles → New Role.**
2. Fill in a name and description.
3. Set the boolean flags (`can_chat`, `can_view_tethers`, etc.).
4. Use the allow-list sections to add databases, documentation sources, and MCP tools.
5. Set `max_row_limit`.
6. Save.

**Assigning roles to resources** (reports, dashboards, tethers): edit the
resource and add the role to its **Allowed roles** field.

**Deleting a role:** only possible when no users are assigned. The delete button
shows a warning if users are attached — reassign them first.

## Managing agents

**Console → Agents**

At most one agent is **Active** at a time. All chat, documentation generation,
dashboard generation, and tether generation use the active agent.

| Action | Notes |
|---|---|
| **Add** | Choose agent type, paste credential, optionally override `service_url`. |
| **Set active** | Check the **Active** checkbox and save. The previous active agent is deactivated automatically. |
| **Edit system prompt** | The system prompt is sent with every request. Changes take effect on the next message — no restart. For CLI-backed agents the prompt is also synced to the gateway container as `AGENTS.md` via `/update-agents-md`. |
| **Test** | Use the **Test** button to verify the gateway is reachable and the credential is valid. |

Switching the active agent is a **zero-downtime database change** — all agent
gateway containers run by default, so there is no container start-up delay.

## Managing database connections

**Console → Databases**

| Action | Notes |
|---|---|
| **Add** | Fill in engine, host, port, database, credentials, and description. |
| **Test** | Verifies connectivity and reports the server version. Uses the same code path as the MCP server. |
| **Edit** | Any field change is picked up by the MCP server on the next tool call — no restart required. |
| **Deactivate** | Set `is_active = False`. The connection disappears from role allow-lists and is no longer queryable. |
| **Delete** | Removes the connection. Existing `QueryAuditLog` entries retain a `null` `database` FK. |

Best practice: always use a **read-only database account** and leave **Read-only**
checked. See [[TetherDust Documentation/5. Security/2. Read-Only Enforcement.md|Read-Only Enforcement]].

## Managing documentation sources

**Console → Docs**

Documentation sources map to folders under `documentations/` on disk.

| Action | Notes |
|---|---|
| **Auto-discovery** | On every startup, `docker-entrypoint.sh` scans `documentations/` and creates/deactivates `DocumentationSource` records to match the filesystem. |
| **Add manually** | Use **Add Source** if you need to point at a folder with non-default `file_patterns` or a custom description. |
| **Generate with AI** | The **Generate** button on any source opens the AI documentation generator — it builds markdown files in the source's folder using the active agent. See [[TetherDust Documentation/2. Features/3. Docs.md|Docs]] for details. |
| **Edit** | Change description, doc type, file patterns, or active state. |

The `documentations/` directory is mounted as a volume in both the `mcp` and
`web` containers — changes on disk are visible immediately, with no restart.

## Reports and dashboards

**Reports** are SQL queries that run on a schedule and deliver results by email
or download. Managed under **Console → Reports**.

**Dashboards** are collections of AI-generated d3.js charts over live SQL.
Managed under **Console → Dashboards**.

Both features grant access per-role via an **Allowed roles** field on each
resource. Add a role to the field to grant that role's users access.

For full details see [[TetherDust Documentation/2. Features/7. Reports.md|Reports]]
and [[TetherDust Documentation/2. Features/8. Dashboards.md|Dashboards]].

## Monitoring activity

**Console → Home** shows a real-time activity summary:

- Query volume (24 h vs. prior 24 h) and failure rate.
- Per-database query breakdown for the past 7 days.
- 8 most recent queries with user, database, and outcome.

**Console → Audit** has the full logs:

- **Query Log** — every `query_database` call, filterable by success/failure. Last 200 entries.
- **Doc Generation Logs** — every AI documentation run, with full agent output and prompt. Filterable by status.
- **Chart Generation Logs** — every AI chart/dashboard run, with full agent output.

See [[TetherDust Documentation/2. Features/12. Audit Log.md|Audit Log]] for field-level details.

## Restarting services

Most configuration changes take effect without a restart:

| Change | Restart needed? |
|---|---|
| Active agent or system prompt | ❌ No — next request uses new config |
| Database connection added/edited | ❌ No — MCP server reads DB on each call |
| Role or user permissions | ❌ No — evaluated on each request |
| New documentation source folder added to disk | ❌ No — auto-discovered on next startup; manual DB entry works immediately |
| Environment variables (`.env`) | ✅ Yes — `docker compose up -d` (or `docker compose restart <service>`) |
| Code changes (Python, templates) | ✅ Dev mode: auto-reload. Production: `docker compose up --build` |

To restart a single service without rebuilding:

```bash
docker compose restart web
docker compose restart mcp
```

## Upgrading

1. Pull the latest code:
   ```bash
   git pull
   ```

2. Rebuild and restart:
   ```bash
   docker compose up --build -d
   ```

   On startup, `docker-entrypoint.sh` runs `manage.py migrate` automatically —
   database migrations are applied before the app starts serving requests.

3. Verify the new version is running:
   ```bash
   docker compose ps
   docker compose logs web --tail 20
   ```

> TetherDust is under active development. Backwards compatibility and
> data-migration guarantees are not yet in place — review release notes before
> upgrading in environments with persistent data.
