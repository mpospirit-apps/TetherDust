# Agent Implementations — Overview

TetherDust answers natural-language questions by routing them to an **LLM
agent** that calls **MCP tools** to query databases, search documentation, and
build charts/dashboards/tethers. This section documents the ways an LLM agent
can be plugged into TetherDust and how each one works.

**There are five ways to integrate an LLM agent with TetherDust:**

| # | Integration | New Docker container? | External API? | Cost model |
|---|---|---|---|---|
| 1 | [[TetherDust Documentation/3. Agent Integrations/2. CLI Tool with Auth Token.md]] | ✅ Yes | Provider subscription (ChatGPT Plus/Pro · Claude Pro/Max) | Flat |
| 2 | [[TetherDust Documentation/3. Agent Integrations/3. CLI Tool with API Key.md]] | ✅ Yes | Provider pay-per-token | Per token |
| 3 | [[TetherDust Documentation/3. Agent Integrations/4. Direct API Agent.md]] | ❌ No | Provider pay-per-token | Per token |
| 4 | [[TetherDust Documentation/3. Agent Integrations/5. Local LLM with Ollama.md]] | ✅ Yes | None (self-hosted) | Hardware |
| 5 | [[TetherDust Documentation/3. Agent Integrations/6. Gateway (OpenRouter).md]] | ❌ No | Gateway pay-per-token | Per token + markup |

Each option has (or will have) its own page in this section with the full
mechanics. The rest of this overview covers what they all share: the agent
abstraction they extend, the access-control layer they inherit, and the
streaming contract they emit.

## What every integration shares

Regardless of how the model is hosted or billed, every integration plugs into
the same three seams.

### 1. The agent abstraction

- **`BaseAgent`** (`core/agents/base.py`) is the transport-agnostic contract.
  Every integration implements `chat()` — an async generator that streams
  response chunks — plus `supports_mcp()` and `get_name()`. The `chat()`
  signature takes the role-based access inputs (`allowed_tools`,
  `allowed_databases`, `allowed_doc_sources`, `max_row_limit`,
  `custom_mcp_servers`) that any agent needs, and makes no assumption about
  subprocesses, containers, or credential files.
- **`AgentConfiguration`** (`core/models/agent.py`) is the admin-editable,
  database-persisted config for an agent: its `agent_type`, `system_prompt`,
  credentials, and `service_url`. Exactly one row is `is_active` at a time.
- **`get_agent()`** (`core/agents/__init__.py`) is the factory: it reads the
  active `AgentConfiguration` and returns the matching agent instance. New
  integrations register their class here against a new `agent_type`.

Because `get_agent()` reads the active config fresh on every request, switching
the integration in use is a **database change, not a redeploy** — it takes
effect on the next message.

### 2. MCP access control (token filtering)

Role-based access control — which tools, databases, and doc sources a user may
reach, and the row limit on queries — is enforced at the **MCP server**, not
inside the agent. Django computes the restrictions, registers a per-request
filter token (`core/agents/mcp_filter.py`), and hands the agent a tokenized MCP
URL (`http://tdmcp:8001/mcp/<token>`). The MCP server extracts the token and
hides/blocks disallowed tools and data.

This registration helper is **agent-agnostic**: any integration reuses
`register_filter` / `tokenized_mcp_url` / `clear_filter` instead of
re-implementing the handshake, so the same role restrictions apply no matter
which agent serves the request.

### 3. The stream protocol

Agents emit a flat string stream (`core/agents/stream.py`) where structured
events are framed by NUL-prefixed markers — `\x00TOOL:<name>`,
`\x00RESPONSE:<text>`, `\x00THINKING:<text>`, and plain text for partial tokens.
The WebSocket consumer parses these with `parse_chunk`. Any integration emits
the same markers, so the UI behaves identically across backends.

## The five options at a glance

1. **CLI tool with auth token** — a provider's coding CLI runs in its own
   container, authenticated with a subscription session credential. Two providers
   are wired: **Codex** (ChatGPT Plus/Pro) and **Claude Code** (Claude Pro/Max).
   Flat cost; good for high volume. → [[TetherDust Documentation/3. Agent Integrations/2. CLI Tool with Auth Token.md|details]]

2. **CLI tool with API key** — the same container-wrapped-CLI architecture, but
   authenticated with a per-token provider API key passed as an environment
   variable instead of a subscription credential. Supports multiple CLIs (Codex,
   Claude Code, Gemini CLI), each in its own container.
   → [[TetherDust Documentation/3. Agent Integrations/3. CLI Tool with API Key.md|details]]

3. **Direct API agent** — no extra container. The agentic tool-call loop runs
   inside Django using an async HTTP client against the provider's API directly
   (OpenAI, Anthropic, Gemini, Azure, Bedrock, or any compatible endpoint).
   Lower operational complexity and full control over the loop.
   → [[TetherDust Documentation/3. Agent Integrations/4. Direct API Agent.md|details]]

4. **Local LLM with Ollama** — a self-hosted Ollama container exposes an
   OpenAI-compatible API for locally hosted models. No external calls; works
   air-gapped. Reuses the Direct-API agent by changing only the base URL and
   model. Requires capable hardware and tool-call-supporting models.
   → [[TetherDust Documentation/3. Agent Integrations/5. Local LLM with Ollama.md|details]]

5. **Gateway / OpenRouter** — a single OpenAI-compatible gateway routes to many
   providers behind one API key, with the model chosen as a config string. One
   agent class covers every provider, at a small per-token markup.
   → [[TetherDust Documentation/3. Agent Integrations/6. Gateway (OpenRouter).md|details]]

## Adding a new integration

The general recipe for wiring in a new agent type:

1. Implement an agent class extending `BaseAgent` (`chat()`, `supports_mcp()`,
   `get_name()`), consuming the tokenized MCP URL for access control rather than
   re-implementing filter registration.
2. Register it in the `get_agent()` factory against a new `agent_type`.
3. Add the new `agent_type` to `AgentConfiguration.AGENT_TYPE_CHOICES` and
   create a migration.
4. For container-backed CLIs, add a `docker/<name>/` service (a FastAPI wrapper
   + Dockerfile) and a Docker Compose entry; Direct-API options skip this.

The per-option pages describe the specifics for each approach.
